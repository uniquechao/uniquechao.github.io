GitLab的使用规范

定位

巴拉巴拉巴拉巴拉。。。。。

细则管理

1 Member 的管理

下面操作需要找IT的管理员完成

  • 人员的添加
  • 人员的转移
  • 人员的离职
  • Group的创建

2 Group 的管理

  • Group的粒度以实际一个独立完整的产品
  • Group内的人员
  • Group的Master 和Ower 应为产品实际的技术负责人担任。

3 Project 的管理

  • Project 应当属于一个Group
  • 个人实验性的Project可以放到个人域下，其他的属于产品的Project要放到Group下。
  • Project是以一个独立的开发工程项目
    • 不能提交 .git、.svn、.idea 等本地配置文件
    • 不能提交编译中间文件
  • 在所有的Project中需要有以下文件。

  • Readme 包括下面几项

    • 用于说明当前Project的功能、适用场景、属于的产品名
    • 实现的说明 或 Readmine的（wiki或issue）地址。
    • 配置运行方法

  • ChangeLog

    • 记录主要的更改点日期等

4 分支的管理

  • 分支应该分为：Master 、Develop 、Release_xx 、Feature_xxx,Hotfix_xxx
  • Master 分支
    • 不能直接在Master分支上直接开发，下面几种情况可以除外
      • Project 开始
      • 二次开发的小任务
    • Master的合并
  • 及时的删除临时本地和远程分支（Master 和Develop之外的）

5 Release 版本的管理

6 commit message

  • 尽量简短,言简意赅 （ short (50 chars or less) summary ）
  • 使用前缀说明更改的类型：Add 、 Fix、Modify
    可以看这里：http://tbaggery.com/2008/04/19/a-note-about-git-commit-messages.html

7 日常的操作

  • 每天及时的同步origin和本地
  • 保证Master 和Develop 是随时可用
